# Test Findings — Constructs With Uncertain or Missing SBsql Support

This file is part of the example/test database **deliverable**: it reports
places where the documented command surface could not be confirmed against the
SBsql syntax reference at authoring time. These are candidate errors. The
statements are left in the test scripts on purpose — running
`run_example_database.sh` confirms or refutes each one, and the per-script
`logs/*.err.log` plus `logs/errors_summary.log` capture the engine's verdict.

Fixes belong in the engine/parser (a separate effort), not in these scripts.

Legend:
- **USED** — the construct is present in a script and will be exercised.
- **OMITTED** — no documented form was found, so the surface is *not* exercised
  (a coverage gap); listed so support can be added and the test extended.

---

## A. Statements requiring descriptor-bound objects with no documented CREATE DDL

The multimodel statement forms are documented (`multimodel_statements.md`), but
they operate on descriptor-bound objects (vector index/descriptor, search index,
graph, key-value space, document container) whose **creation** DDL is not present
in the SBsql syntax reference. `34_multimodel.sql` exercises the statements
against named objects; with no way to create those objects from the documented
command set, each statement is expected to error (missing object and/or
unsupported statement). **Either outcome is a finding to report.**

| Statement (file `34_multimodel.sql`) | Target object | Status |
| --- | --- | --- |
| `document query` / `document patch` | document container (`app.catalog.products` has a `document` column; unclear if a relational table is a valid target) | USED |
| `vector search` / `vector rerank` (incl. `WITH` form) | `app.product_embedding` — no documented `CREATE` for a vector index/descriptor object | USED |
| `search ... for ...` | `app.product_search` — no documented `CREATE` for a search index object | USED |
| `graph match` / `graph traverse` | `app.customer_graph` — no documented `CREATE` for a graph object | USED |
| `kv get` / `kv put` / `kv scan` | `app.session_cache` — no documented `CREATE` for a key-value space | USED |
| `timeseries query` / `timeseries gapfill` | `ts.sensor_reading` (relational table; unclear if a time-series descriptor is required) | USED |

SQL-callable functions on the same data **are** confirmed and exercised:
`json_value` / `json_exists` (`sb_json`), `cosine_distance` / `hamming_distance`
(`sb_vector`), and standard SQL windowing/aggregates over `ts.sensor_reading`.

## D. Conversion-matrix surfaces not exercised (need named functions / literal syntax)

`37_casts.sql` covers every conversion the matrix marks supported via a bare
`cast`/`try_cast`. The following are documented as supported but could not be
expressed without syntax the data-type pages do not ground, so they are
`OMITTED` (positive coverage gaps) — and likewise their refusal counterparts in
`expected_refusals/cast_refusals.sql`:

| Conversion | Reason omitted |
| --- | --- |
| `text` ↔ `bytea`/binary | Matrix requires an encoding/decode *rule*; the admitted surface is the named functions `encode`/`decode`, not a bare cast. |
| `binary(16)` → `uuid` | Supported per matrix, but constructing a 16-byte `binary(16)` argument needs a binary literal / `decode`, which is a named function, not a cast literal. |
| `array<T>` element conversion | No portable array-literal construction syntax (e.g. `ARRAY[...]`) is shown in the reviewed `data_types` pages. |
| `vector` element-type change / dimension mismatch | No vector-literal construction or element-cast syntax is grounded in the reviewed pages (only `vector search ... using :param`). |
| binary-with-invalid-bytes → `text` (refusal) | Same binary-literal dependency as above. |
| protected-material release (refusal) | No protected-material column/function exists in the example schema; deferred to a security-surface test. |

These belong in the conversion proof suite once the function/literal syntax is
confirmed; the cast scripts note each one inline.
